# Восстановление осиротевших кампаний при старте (Фаза 2: durability)

**Дата:** 2026-06-21
**Статус:** утверждён (брейншторм)
**Область:** `internal/store`, `cmd/server`

## Проблема

`BackgroundRunner` выполняет прогон кампании в горутине in-memory. Если сервис падает или
перезапускается во время прогона, кампания остаётся в статусе `pending` или `running` навсегда:
горутина умерла, а `cmd/server/main.go` при старте такие записи не разбирает. В UI они вечно
показываются «в работе».

Это единственный незакрытый durability-гэп Фазы 2. Остальное уже реализовано:
- LLM-ретраи с экспоненциальным backoff на 429/5xx (`internal/llm/openai.go`, покрыто тестом);
- таймаут прогона `RUN_TIMEOUT` (`context.WithTimeout`);
- graceful shutdown через `runner.Drain()`.

## Решение

При старте сервиса (до приёма трафика) пометить все осиротевшие кампании (`pending`/`running`)
как `failed` с понятным сообщением. Пользователь сам перезапустит кампанию, если нужно. Так мы
избегаем сюрприз-расходов на LLM и дублирования работы (у пайплайна нет идемпотентности).

(Отклонены альтернативы: ленивое восстановление на чтении — не чистит список надёжно;
периодический sweeper — оверкилл для задачи «пережить рестарт», YAGNI.)

## Архитектура

Граница чистая: метод стора инкапсулирует SQL, `main` дёргает его один раз на старте.

### `internal/store/store.go` — новый метод

```go
// RecoverInterrupted помечает осиротевшие после рестарта кампании (pending/running)
// как failed. Возвращает число восстановленных. Идемпотентен.
func (s *Store) RecoverInterrupted(ctx context.Context) (int64, error)
```

Внутри — один атомарный апдейт:

```sql
UPDATE campaigns
SET status='failed', error='прервано рестартом сервиса', updated_at=now()
WHERE status IN ('pending','running')
```

Возвращает `tag.RowsAffected()` из результата `pool.Exec`.

### `cmd/server/main.go` — вызов на старте

Сразу после `st := store.New(pool)` (после `store.Migrate`, до запуска HTTP-сервера):

```go
n, err := st.RecoverInterrupted(baseCtx)
if err != nil {
    logger.Error("recover interrupted", "err", err)
    os.Exit(1)
}
if n > 0 {
    logger.Info("recovered interrupted campaigns", "count", n)
}
```

Происходит до приёма трафика — гонок с новыми прогонами нет.

## Обработка ошибок

- `RecoverInterrupted` прокидывает ошибку наверх (как остальные методы стора).
- В `main.go` ошибка восстановления фатальна на старте (`os.Exit(1)`), по аналогии с
  `store.Migrate`: если БД недоступна на старте — сервису всё равно нельзя работать.
- Идемпотентность: повторный вызов на чистом состоянии затрагивает 0 строк и безвреден.

## Тестирование

Следуя существующему харнесу `//go:build integration` (реальный Postgres через `DATABASE_URL`,
`newTestStore` мигрирует схему). Новый тест в `internal/store/store_test.go`:

1. Создать кампанию (`Create` → `pending`); вторую перевести в `running` (`MarkRunning`);
   третью — в `done` (`Complete` с фиктивным результатом).
2. Вызвать `RecoverInterrupted` → ожидаем `count == 2`.
3. Через `Get` проверить: первые две → `failed` с текстом «прервано рестартом сервиса»;
   `done`-кампания не тронута.
4. Повторный вызов → `count == 0` (идемпотентность).

Юнит-тестов без БД не добавляем (логика — чистый SQL). Прогон:
`go test -tags=integration ./internal/store` с поднятым Postgres. Обычный `go test ./...`
остаётся зелёным без изменений.

## Вне области (YAGNI)

- Авто-перезапуск осиротевших кампаний.
- Heartbeat / `started_at` для детекции зависших running по таймауту в работающем сервисе.
- Периодический sweeper.

## Рабочий процесс

Фичу делаем на отдельной ветке (например, `feat/recover-interrupted`), затем тесты → merge в `main`.
